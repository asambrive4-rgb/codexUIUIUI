# codexUIUIUI Design System Guidelines

> 문서 상태: Draft v1  
> 대상 제품: **codexUIUIUI**  
> 대상 플랫폼: Windows 데스크톱  
> UI 기술 방향: WPF + MVVM  
> 제품 성격: Codex 로컬 프로필 관리 및 전환 유틸리티  
> 기본 테마: Light Pastel Navy Blue  
> MVP 정책: 단일 라이트 테마 우선 구현

---

## 0. 문서 목적

이 문서는 `codexUIUIUI`의 화면을 일관된 방식으로 설계하고 구현하기 위한 시각·인터랙션 기준을 정의한다.

기존 디자인 가이드에서 다음 원칙은 유지한다.

- 부드럽고 정돈된 카드형 화면
- 명확한 Hover, Pressed, Focused, Disabled 상태
- 지나치게 장식적이지 않은 친근한 인상
- 가독성과 기능 명확성을 우선하는 UI
- WPF에서 재사용 가능한 디자인 토큰 중심 구현

다음 요소는 제거한다.

- 오리, 마스코트, 캐릭터 이미지
- DeckDuck 관련 에셋과 자리표시자
- DeckDeckDeck 제품명과 브랜드 표현
- 덱, 텐키, Markdown 카드 등 기존 제품 기능에 대한 언급
- 기존 제품의 기능 구조에 종속된 화면 규칙

이 문서는 `codexUIUIUI`의 프로필 생성, 최초 로그인, 실행, 전환, 삭제 흐름에 맞게 최적화한다.

## 1. 시각 시스템 방향

### 1.1 핵심 인상

`codexUIUIUI`는 다음과 같은 인상을 목표로 한다.

> **차분함 + 신뢰감 + 가벼움 + 명확한 상태 표현**

계정 인증 환경을 다루는 도구이므로 귀엽거나 화려한 느낌보다, 사용자가 현재 상태와 다음 동작을 쉽게 이해할 수 있는 안정적인 인상이 중요하다.

동시에 개발자용 설정 도구처럼 지나치게 딱딱해 보이지 않도록 연한 파스텔 네이비 블루를 사용한다.

### 1.2 시각 키워드

- Soft
- Calm
- Trustworthy
- Local
- Focused
- Minimal
- Windows-native

### 1.3 우선순위

| 우선순위 | 원칙 | 설명 |
|---:|---|---|
| 1 | 상태 명확성 | 현재 프로필, 실행 중, 전환 중, 오류 상태가 즉시 구분되어야 한다 |
| 2 | 가독성 | 텍스트와 버튼 대비가 충분해야 한다 |
| 3 | 안전한 행동 | 삭제, 강제 종료처럼 위험한 행동은 일반 행동과 명확히 구분한다 |
| 4 | 일관성 | 같은 의미의 상태와 버튼은 항상 같은 색과 형태를 사용한다 |
| 5 | 부드러운 인상 | 파스텔 네이비와 넉넉한 여백으로 긴장감을 낮춘다 |
| 6 | 장식 절제 | 장식은 기능과 상태 인지를 방해하지 않는 범위에서만 사용한다 |

## 2. 브랜드 및 제품명 표기

### 2.1 공식 앱 이름

```text
codexUIUIUI
```

제품명은 문서, 화면 제목, 설치 파일 정보에서 위 표기를 그대로 사용한다.

### 2.2 권장 부제

한국어:

```text
Codex 로컬 프로필 관리 도구
```

영문:

```text
Local profile manager for Codex
```

### 2.3 표기 규칙

- 제품명은 항상 `codexUIUIUI`로 표기한다.
- `CodexUIUIUI`, `codex-ui-ui-ui`, `Codex UI Manager` 등으로 임의 변경하지 않는다.
- 앱 내부에서는 실제 OpenAI 계정을 직접 관리하는 것처럼 보이는 표현을 피한다.
- 주 UI 용어는 `계정`보다 `프로필`을 우선한다.
- 앱 정보 화면에는 비공식 로컬 유틸리티임을 명시한다.

권장 안내 문구:

```text
codexUIUIUI는 계정별로 분리된 Codex 로컬 프로필을
관리하고 실행하기 위한 Windows 유틸리티입니다.

OpenAI의 공식 제품이 아닙니다.
```

## 3. 컬러 시스템

### 3.1 기본 방향

기본 컬러 방향은 다음과 같다.

> **Cool Background + Pastel Navy Surface + Deep Navy Action**

파스텔 톤은 배경과 선택 면에 사용하고, 주요 버튼과 핵심 텍스트에는 충분히 진한 네이비를 사용한다.

연한 색만 사용하면 버튼과 텍스트의 가독성이 떨어질 수 있으므로, 시각적으로 부드럽더라도 기능 요소의 대비는 유지한다.

### 3.2 Core Palette

| Token | Hex | 용도 |
|---|---:|---|
| `bg.app` | `#F4F7FB` | 앱 전체 배경 |
| `bg.surface` | `#FFFFFF` | 카드, 패널, 모달 배경 |
| `bg.surface.soft` | `#EAF0F7` | 보조 영역, Hover 배경 |
| `brand.soft` | `#D8E4F2` | 선택 영역, 활성 프로필의 연한 면 |
| `brand.pastel` | `#B9CBE0` | 보조 강조, 상태 배지 |
| `brand.primary` | `#6F88A6` | 아이콘, 보조 버튼, 선택 강조 |
| `brand.strong` | `#506A89` | 주요 버튼, Focus |
| `brand.deep` | `#30465F` | 제목, 강한 강조, 네비게이션 |
| `text.primary` | `#26384D` | 기본 본문 및 제목 |
| `text.secondary` | `#627389` | 보조 설명, 메타데이터 |
| `text.muted` | `#8B98A8` | 비활성 보조 정보 |
| `line.soft` | `#D5DEE9` | 기본 테두리 및 구분선 |
| `line.strong` | `#AEBFD2` | Hover, 선택 상태 테두리 |
| `focus.ring` | `#52779F` | 키보드 Focus ring |
| `overlay.scrim` | `#5226384D` | 모달 뒤 반투명 배경 |

### 3.3 Semantic Palette

상태색은 브랜드색과 별도로 의미를 고정한다.

| Token | Hex | 용도 |
|---|---:|---|
| `state.success` | `#557E6D` | 실행 성공, 정상 상태 |
| `state.success.soft` | `#E5F0EB` | 성공 메시지 배경 |
| `state.warning` | `#8A704A` | 저장 대기, 주의 필요 |
| `state.warning.soft` | `#F6F0E4` | 경고 메시지 배경 |
| `state.danger` | `#A65360` | 삭제, 강제 종료, 치명적 오류 |
| `state.danger.hover` | `#8E4550` | 위험 버튼 Hover |
| `state.danger.soft` | `#F8E8EB` | 삭제 확인, 오류 배경 |
| `state.info` | `#55789D` | 일반 안내 |
| `state.info.soft` | `#E6EEF7` | 안내 배경 |
| `state.disabled` | `#A4AFBC` | 비활성 텍스트와 아이콘 |
| `state.disabled.bg` | `#EEF1F5` | 비활성 배경 |

### 3.4 컬러 사용 규칙

- 화면 전체에 진한 네이비를 넓게 사용하지 않는다.
- 진한 네이비는 제목, 주요 버튼, 선택된 핵심 요소에 제한한다.
- 활성 프로필은 `brand.soft` 배경과 `brand.strong` 테두리 또는 상태 배지로 구분한다.
- 삭제 버튼은 평상시 과도하게 강조하지 않는다.
- 위험 동작을 확정하는 순간에만 `state.danger`를 사용한다.
- 성공과 오류를 색상만으로 전달하지 않고 아이콘과 문구를 함께 사용한다.
- 흰색 텍스트를 사용하는 버튼은 `brand.strong` 이상으로 충분히 진한 배경을 사용한다.

## 4. Theme Policy

### 4.1 MVP 정책

MVP에서는 라이트 테마 하나만 구현한다.

| 항목 | 정책 |
|---|---|
| 기본 테마 | Light Pastel Navy |
| 다크 모드 | MVP 제외 |
| 사용자 색상 선택 | MVP 제외 |
| Windows 시스템 테마 연동 | 추후 검토 |
| 고대비 모드 | 기본 Windows 접근성 동작을 방해하지 않음 |

### 4.2 추후 확장

다크 모드를 추가할 경우 기존 색상값을 직접 덮어쓰지 않고 동일한 토큰 이름에 다크 팔레트를 연결한다.

예:

```text
Light: bg.app = #F4F7FB
Dark:  bg.app = 별도 다크 네이비 값
```

화면 코드에서 Hex 값을 직접 사용하지 않고 토큰을 참조해야 테마 확장이 가능하다.

## 5. Layout System

### 5.1 기본 화면 구조

메인 화면은 다음 영역으로 구성한다.

```text
┌────────────────────────────────────────────┐
│ App Header                                 │
│ codexUIUIUI                    설정        │
├────────────────────────────────────────────┤
│ Status Summary                             │
│ 현재 실행 프로필 또는 Codex 상태           │
├────────────────────────────────────────────┤
│ Profile List                               │
│ 프로필 카드 반복                           │
│                                            │
│ 새 프로필 추가                             │
├────────────────────────────────────────────┤
│ Optional Status / Toast Area               │
└────────────────────────────────────────────┘
```

### 5.2 권장 크기

| 항목 | 권장값 |
|---|---:|
| 기본 창 너비 | `720px ~ 860px` |
| 기본 창 높이 | `560px ~ 720px` |
| 최소 창 너비 | `620px` |
| 외부 여백 | `24px` |
| 섹션 간격 | `24px ~ 32px` |
| 카드 간격 | `12px` |
| 카드 내부 여백 | `16px ~ 20px` |
| 작은 요소 간격 | `6px ~ 8px` |
| 일반 요소 간격 | `12px ~ 16px` |

### 5.3 반응형 원칙

WPF 데스크톱 앱이므로 모바일 반응형은 필요하지 않지만, 창 크기 변경에 대응해야 한다.

- 좁은 화면에서는 프로필 카드의 버튼을 오른쪽 고정 대신 아래쪽으로 배치할 수 있다.
- 프로필 이름이 길면 한 줄 말줄임을 사용한다.
- 버튼 문구는 잘리지 않아야 한다.
- 최소 너비 이하로 줄이지 못하도록 제한한다.
- 스크롤은 프로필 목록 영역에만 적용하는 것을 우선한다.

## 6. Shape and Elevation

### 6.1 형태 방향

전체 형태는 **Soft Control Surface**를 기준으로 한다.

이는 딱딱한 관리 도구와 장식적인 소비자 앱의 중간 형태다.

| 요소 | 권장 방향 |
|---|---|
| 모서리 | 부드러운 둥근 사각형 |
| 그림자 | 매우 약한 elevation |
| 테두리 | 기본은 연하게, 상태 변화 시 명확하게 |
| 내부 여백 | 충분히 넓게 |
| 아이콘 | 단순한 선형 아이콘 |
| 장식 | 별도 캐릭터나 일러스트 없이 최소화 |

### 6.2 권장 수치

| 요소 | 권장값 |
|---|---:|
| Main panel radius | `18px` |
| Profile card radius | `14px ~ 16px` |
| Button radius | `10px ~ 12px` |
| Input radius | `10px` |
| Modal radius | `18px` |
| Border width | `1px` |
| Focus ring width | `2px` |
| Default shadow blur | `12px ~ 18px` |
| Default shadow opacity | `6% ~ 10%` |
| Hover translation | `-1px` |
| Pressed translation | `1px` |

그림자는 화면을 떠다니게 만들 정도로 강하게 사용하지 않는다.

## 7. Typography

### 7.1 폰트

| 용도 | 권장 |
|---|---|
| 기본 Windows UI | `Segoe UI Variable`, `Segoe UI` |
| 한국어 fallback | `Malgun Gothic` |
| 경로 및 기술 정보 | `Cascadia Mono`, `Consolas` |
| 앱 내 일반 본문 | 시스템 sans-serif |

외부 폰트 설치를 요구하지 않는다.

### 7.2 타입 스케일

| 역할 | 크기 | 굵기 |
|---|---:|---:|
| App title | `24px` | 700 |
| Page title | `22px` | 700 |
| Section title | `18px` | 650~700 |
| Profile title | `16px` | 600 |
| Body | `14px` | 400 |
| Button | `14px` | 600 |
| Caption | `12px` | 400 |
| Status badge | `11px ~ 12px` | 600 |

### 7.3 텍스트 원칙

- 버튼 문구는 동작 결과가 예상되도록 쓴다.
- `확인`, `진행`처럼 의미가 약한 표현보다 구체적인 문구를 사용한다.
- 실제 계정을 확인하지 못하는 상태에서 이메일 또는 계정명을 단정해 표시하지 않는다.
- `현재 계정`보다 `현재 실행 프로필`을 사용한다.
- 위험 동작에는 결과와 복구 불가능성을 명시한다.
- 기술 오류는 사용자 문구와 상세 정보 영역을 분리한다.

## 8. Iconography

### 8.1 방향

- 단순한 선형 아이콘을 사용한다.
- 동일한 아이콘 세트 안에서 선택한다.
- 16px, 20px, 24px 기준 크기를 사용한다.
- 아이콘만으로 의미를 전달하지 않고 중요 행동에는 텍스트를 함께 둔다.

### 8.2 권장 아이콘 의미

| 기능 | 권장 아이콘 |
|---|---|
| 프로필 | User 또는 Folder |
| 실행 | Play 또는 ArrowRight |
| 전환 | SwitchHorizontal 또는 Refresh |
| 추가 | Plus |
| 삭제 | Trash |
| 설정 | Settings |
| 정보 | Info |
| 경고 | WarningTriangle |
| 성공 | CheckCircle |
| 오류 | ErrorCircle |

### 8.3 피해야 할 것

- 제품과 관계없는 캐릭터형 아이콘
- 색상이 과도하게 많은 아이콘
- Filled와 Outline 스타일 혼용
- 삭제 아이콘을 일반 탐색 아이콘과 같은 색으로 표시
- 의미가 불분명한 추상 아이콘만 사용

## 9. Profile Card Component

### 9.1 역할

프로필 카드는 사용자가 관리하는 로컬 Codex 프로필 하나를 나타낸다.

카드에 표시할 수 있는 정보:

- 프로필 이름
- 현재 상태
- 마지막 실행 시각
- 최초 로그인 필요 여부
- 실행 또는 전환 버튼
- 추가 작업 메뉴

카드에서 표시하지 않는 정보:

- 인증 토큰
- `auth.json` 내용
- 비밀번호
- 확실하게 확인되지 않은 실제 OpenAI 이메일
- 긴 파일 경로 전체

### 9.2 기본 구조

```text
┌────────────────────────────────────────────┐
│ [Icon] Work                    [사용 중]   │
│        마지막 실행: 오늘 오후 2:30         │
│                                            │
│                      [Codex 열기] [⋯]      │
└────────────────────────────────────────────┘
```

### 9.3 카드 상태

| 상태 | 표현 | 주요 동작 |
|---|---|---|
| Default | 흰색 면, 연한 테두리 | 이 프로필로 실행 |
| Hover | 연한 네이비 면, 진해진 테두리 | 클릭 가능성 표시 |
| Pressed | 그림자 감소, 1px 하강 | 즉각적인 눌림 피드백 |
| Focused | 외부 2px Focus ring | 키보드 위치 표시 |
| Active | `brand.soft` 배경, 상태 배지 | Codex 열기 |
| Switching | 버튼 비활성, 진행 표시 | 중복 전환 방지 |
| Login required | 안내 배지와 로그인 시작 버튼 | 최초 로그인 |
| Disabled | 낮은 대비와 사유 표시 | 실행 불가 |
| Error | 위험색 전체 도배 금지, 오류 문구 표시 | 다시 시도 또는 상세 보기 |

### 9.4 Active 상태

활성 상태는 색상 하나에만 의존하지 않는다.

권장 조합:

- `brand.soft` 배경
- `brand.strong` 테두리
- `사용 중` 또는 `현재 실행 프로필` 배지
- Check 또는 작은 상태 점
- 기본 카드보다 약간 강한 제목

### 9.5 상태별 버튼 문구

| 조건 | 권장 문구 |
|---|---|
| 신규 프로필 | `로그인 시작` |
| Codex 미실행 | `이 프로필로 실행` |
| 다른 프로필 실행 중 | `이 프로필로 전환` |
| 현재 프로필 실행 중 | `Codex 열기` |
| 전환 진행 중 | `전환 중…` |
| 실행 실패 후 | `다시 시도` |

## 10. Empty State

### 10.1 원칙

프로필이 하나도 없을 때는 장식 이미지 없이 핵심 행동을 중심으로 보여준다.

```text
아직 등록된 프로필이 없습니다

Codex에서 사용할 계정별 프로필을 만들어 보세요.
이메일과 비밀번호는 codexUIUIUI에 저장되지 않습니다.

[첫 프로필 만들기]
```

### 10.2 디자인

- 중앙 정렬을 사용하되 지나치게 넓은 빈 공간을 만들지 않는다.
- 연한 `brand.soft` 아이콘 원형 또는 단순 Folder/User 아이콘을 사용할 수 있다.
- 주요 CTA는 하나만 둔다.
- 보안 안내는 짧고 명확하게 표시한다.
- 캐릭터, 마스코트, 장식용 이미지 영역은 두지 않는다.

## 11. Buttons

### 11.1 버튼 계층

| 유형 | 용도 | 스타일 |
|---|---|---|
| Primary | 프로필 실행, 전환, 생성 완료 | `brand.strong` 배경 + 흰색 텍스트 |
| Secondary | 취소, 설정, 상세 보기 | 흰색/연한 면 + 네이비 테두리 |
| Tertiary | 카드 내 부가 동작 | 투명 배경 + 텍스트 |
| Danger | 삭제 확정, 강제 종료 | `state.danger` 배경 + 흰색 텍스트 |
| Ghost Danger | 삭제 메뉴 진입 | 투명 배경 + `state.danger` 텍스트 |

### 11.2 상태

#### Default

- 버튼 역할과 문구가 명확해야 한다.
- Primary 버튼은 한 화면에 너무 많이 두지 않는다.

#### Hover

- 배경을 한 단계 진하게 한다.
- 크기를 크게 확대하지 않는다.

#### Pressed

- `1px` 아래로 이동하거나 그림자를 줄인다.
- 과한 애니메이션 없이 즉시 반응한다.

#### Focused

- 버튼 바깥에 `2px` Focus ring을 표시한다.
- 키보드 사용 시 Hover보다 Focus가 더 명확해야 한다.

#### Disabled

- 클릭할 수 없다는 점이 분명해야 한다.
- 낮은 opacity만 사용하지 말고 색상 토큰도 변경한다.
- 필요한 경우 비활성 사유를 Tooltip으로 제공한다.

### 11.3 최소 크기

| 항목 | 권장값 |
|---|---:|
| 최소 높이 | `40px` |
| 기본 좌우 padding | `16px` |
| 아이콘 버튼 최소 크기 | `36px x 36px` |
| 주요 CTA 최소 너비 | `120px` |

## 13. Status and Feedback

### 13.1 상태 피드백 원칙

사용자가 버튼을 누른 뒤 앱이 현재 무엇을 하는지 알 수 있어야 한다.

계정 전환 과정은 내부적으로 Codex 종료와 재실행을 포함할 수 있으므로 진행 상태를 숨기지 않는다.

### 13.2 전환 진행 표시

권장 단계:

```text
Codex 종료 확인 중…
Work 프로필 준비 중…
Codex 다시 여는 중…
```

단, 모든 내부 기술 단계를 상세하게 노출할 필요는 없다.

간결한 기본 문구:

```text
Work 프로필로 전환하고 있습니다…
```

필요 시 아래에 작은 단계 문구를 표시한다.

### 13.3 Toast

Toast는 짧은 결과 전달에 사용한다.

| 상황 | 문구 예시 |
|---|---|
| 실행 성공 | `Work 프로필로 Codex를 열었습니다.` |
| 전환 성공 | `Work 프로필로 전환했습니다.` |
| 프로필 생성 | `Work 프로필을 만들었습니다.` |
| 삭제 성공 | `Work 프로필과 로컬 데이터를 삭제했습니다.` |
| 일반 오류 | `Codex를 열지 못했습니다. 다시 시도해 주세요.` |

### 13.4 오류 메시지

사용자에게는 해결 가능한 언어를 먼저 보여준다.

권장 구조:

```text
Codex를 종료하지 못했습니다.

Codex에서 진행 중인 작업을 확인한 뒤 다시 시도해 주세요.

[취소] [다시 시도]
```

상세 기술 정보는 접을 수 있는 영역에 둔다.

```text
기술 정보 보기
```

## 14. Modal and Confirmation

### 14.1 기본 모달

- 앱 화면 중앙에 표시한다.
- 뒤 배경에는 `overlay.scrim`을 적용한다.
- 기본 폭은 `420px ~ 520px`를 권장한다.
- 제목, 설명, 행동 버튼 순서를 유지한다.
- 우측 상단 닫기 버튼이 있더라도 하단 취소 버튼을 함께 둔다.

### 14.2 삭제 확인 모달

```text
Work 프로필을 삭제하시겠습니까?

이 프로필의 Codex 로그인 정보, 설정 및 로컬 데이터가
함께 삭제됩니다. 삭제 후에는 복구할 수 없습니다.

[취소] [프로필 및 데이터 삭제]
```

디자인 규칙:

- 모달 전체를 빨간색으로 만들지 않는다.
- 제목 또는 경고 아이콘에 `state.danger`를 사용한다.
- 삭제 확정 버튼만 Danger 스타일로 표시한다.
- 기본 Focus는 `취소`에 두는 것을 권장한다.
- Enter만으로 실수로 삭제되지 않도록 한다.
- Escape로 취소할 수 있어야 한다.

### 14.3 강제 종료 확인

Codex 정상 종료에 실패한 경우:

```text
Codex를 정상적으로 종료하지 못했습니다.

강제 종료하면 Codex에서 저장되지 않은 작업이 사라질 수 있습니다.

[돌아가기] [강제 종료 후 전환]
```

이 버튼도 Danger 스타일을 사용한다.

## 16. Accessibility

### 16.1 키보드

- 모든 버튼과 카드 동작은 Tab으로 접근 가능해야 한다.
- Focus 순서는 화면의 시각적 순서와 일치해야 한다.
- Enter와 Space로 버튼을 실행할 수 있어야 한다.
- Escape로 모달을 닫을 수 있어야 한다.
- 삭제 모달의 초기 Focus는 취소 버튼을 권장한다.

### 16.2 색상

- 상태를 색상만으로 구분하지 않는다.
- 활성 프로필에는 배지, 아이콘 또는 문구를 추가한다.
- 오류에는 오류 아이콘과 문구를 함께 표시한다.
- 비활성 텍스트도 읽을 수 있는 최소 대비를 유지한다.

### 16.3 텍스트와 크기

- Windows 배율 `100%`, `125%`, `150%`, `200%`에서 레이아웃을 확인한다.
- 텍스트를 이미지로 만들지 않는다.
- 고정 높이로 인해 문구가 잘리지 않도록 한다.
- 한국어와 영문 길이 차이를 고려한다.

### 16.4 스크린 리더

중요 컴포넌트에는 접근 가능한 이름을 제공한다.

예:

```text
Work 프로필, 현재 사용 중, Codex 열기
```

아이콘 버튼에는 의미 있는 AutomationProperties.Name을 지정한다.

## 17. Motion

### 17.1 원칙

애니메이션은 상태 이해를 돕는 범위에서만 사용한다.

| 상황 | 권장 |
|---|---|
| Hover | `100ms ~ 150ms` |
| 버튼 Pressed | 즉시 또는 `80ms` |
| 모달 열림 | `150ms ~ 200ms` fade/scale |
| Toast | `150ms` fade |
| 전환 진행 | 작은 indeterminate progress |
| 카드 재정렬 | 필요할 때만 부드럽게 |

### 17.2 피해야 할 것

- 큰 Bounce
- 긴 회전 애니메이션
- 화면 전체 슬라이드
- 계속 반복되는 장식 애니메이션
- 상태보다 시선을 더 끄는 진행 표시

Windows의 애니메이션 감소 설정을 존중할 수 있도록 구현한다.

## 18. WPF Resource Token Guidelines

### 18.1 리소스 구조

```text
Presentation.Wpf/
└── Resources/
    ├── Colors.xaml
    ├── Brushes.xaml
    ├── Typography.xaml
    ├── Spacing.xaml
    ├── Controls/
    │   ├── Buttons.xaml
    │   ├── ProfileCards.xaml
    │   ├── Inputs.xaml
    │   ├── Badges.xaml
    │   ├── Toasts.xaml
    │   └── Dialogs.xaml
    └── Themes/
        └── LightPastelNavy.xaml
```

### 18.2 Color 예시

```xml
<Color x:Key="Color.Background.App">#F4F7FB</Color>
<Color x:Key="Color.Background.Surface">#FFFFFF</Color>
<Color x:Key="Color.Brand.Soft">#D8E4F2</Color>
<Color x:Key="Color.Brand.Primary">#6F88A6</Color>
<Color x:Key="Color.Brand.Strong">#506A89</Color>
<Color x:Key="Color.Brand.Deep">#30465F</Color>
<Color x:Key="Color.Text.Primary">#26384D</Color>
<Color x:Key="Color.Text.Secondary">#627389</Color>
<Color x:Key="Color.Line.Soft">#D5DEE9</Color>
<Color x:Key="Color.Focus.Ring">#52779F</Color>
<Color x:Key="Color.State.Danger">#A65360</Color>
```

### 18.3 Brush 예시

```xml
<SolidColorBrush
    x:Key="Brush.Background.App"
    Color="{StaticResource Color.Background.App}" />

<SolidColorBrush
    x:Key="Brush.Brand.Strong"
    Color="{StaticResource Color.Brand.Strong}" />

<SolidColorBrush
    x:Key="Brush.Text.Primary"
    Color="{StaticResource Color.Text.Primary}" />
```

### 18.4 규칙

- View XAML에 Hex 색상을 직접 입력하지 않는다.
- ViewModel에서 Brush, Color, Thickness 같은 WPF 타입을 만들지 않는다.
- 상태는 ViewModel이 의미 값으로 제공한다.
- View가 상태에 맞는 리소스와 스타일을 선택한다.
- 공통 버튼과 카드 스타일은 ResourceDictionary로 관리한다.

예:

```text
ViewModel:
ProfileState = Active

View:
Active 상태를 감지
→ ActiveProfileCardStyle 적용
```

이는 MVVM과 Clean Architecture의 경계를 유지하기 위한 규칙이다.

## 19. MVVM State Mapping

### 19.1 ViewModel 상태 예시

```text
ProfileItemViewModel
├── Name
├── LastUsedText
├── Status
├── PrimaryActionText
├── IsBusy
├── CanExecute
```


### 19.3 화면 매핑

|---|---|
| `NotAuthenticated` | 로그인 필요 배지, `로그인 시작` |

| `Error` | 오류 문구, `다시 시도` |

ViewModel은 구체적인 Hex 색상이나 WPF 스타일 이름을 알지 않는다.

## 21. Onboarding

### 21.1 첫 실행 안내

MVP에서는 여러 장의 튜토리얼보다 한 화면의 간단한 설명을 권장한다.

```text
codexUIUIUI 시작하기

계정마다 분리된 Codex 프로필을 만들고,
필요할 때 원하는 프로필로 Codex를 실행할 수 있습니다.

• 이메일과 비밀번호는 저장하지 않습니다.
• 최초 로그인은 Codex의 기존 로그인 절차를 사용합니다.
• ChatGPT 웹 세션에 영향을 주지 않는 것을 목표로 합니다.

[첫 프로필 만들기]
```

### 21.2 주의

웹 세션 비간섭이 기술적으로 검증되기 전에는 확정적인 보장 문구를 사용하지 않는다.

검증 전:

```text
ChatGPT 웹 세션에 영향을 주지 않는 방식으로 동작하는지 확인합니다.
```

검증 완료 후:

```text
ChatGPT 웹 세션을 유지한 채 Codex 프로필을 전환합니다.
```

## 22. Security-related UI

### 22.1 보안 안내

보안 문구는 과도한 공포감을 주지 않되 정확해야 한다.

권장:

```text
codexUIUIUI는 이메일과 비밀번호를 저장하지 않습니다.
Codex 인증 데이터는 각 프로필의 로컬 폴더에 저장됩니다.
```

### 22.2 민감 정보 표시 금지

UI와 로그에 다음 내용을 표시하지 않는다.

- 인증 토큰
- 인증 파일 원문
- 브라우저 쿠키
- 전체 환경변수
- 민감한 사용자 경로가 포함된 상세 오류
- 계정 이메일이라고 확인되지 않은 값

### 22.3 앱 데이터 폴더

설정에서 `앱 데이터 폴더 열기` 기능을 제공할 경우, 사용자가 인증 파일을 임의로 수정하지 않도록 경고 문구를 표시할 수 있다.

```text
이 폴더에는 Codex 로그인 상태와 관련된 파일이 포함될 수 있습니다.
파일을 공유하거나 직접 수정하지 마세요.
```

## 23. Content Guidelines

### 23.1 용어

| 피할 표현 | 권장 표현 |
|---|---|
| 계정 추가 | 프로필 추가 |
| 계정 삭제 | 프로필 삭제 |
| 현재 계정 | 현재 실행 프로필 |
| 계정 저장 | 인증 환경 저장 |
| 로그인 정보 가져오기 | Codex 로그인 시작 |
| 계정 변경 | 프로필 전환 |
| 비밀번호 보관 | 사용하지 않음 |

### 23.2 문장 스타일

- 짧고 직접적으로 쓴다.
- 한 문장에는 하나의 행동만 설명한다.
- 사용자에게 내부 구현 용어를 강요하지 않는다.
- 오류 문구에는 가능한 다음 행동을 포함한다.
- `CODEX_HOME`, `auth.json` 같은 용어는 기본 화면에 노출하지 않는다.
- 기술 용어는 설정의 상세 정보나 진단 화면에서만 사용한다.

### 23.3 예시

좋음:

```text
Work 프로필로 전환했습니다.
```

피함:

```text
CODEX_HOME 환경변수 변경 및 프로세스 재기동이 완료되었습니다.
```

좋음:

```text
Codex를 종료하지 못했습니다. 진행 중인 작업을 확인한 뒤 다시 시도해 주세요.
```

피함:

```text
Process termination failed with exit code 1.
```

## 24. Design QA Checklist

### 24.1 공통

- [ ] 제품명이 `codexUIUIUI`로 정확히 표기되는가?
- [ ] 기존 제품명, 오리, 캐릭터, 덱 관련 요소가 남아 있지 않은가?
- [ ] 화면에 직접 입력한 Hex 색상이 없는가?
- [ ] 주요 버튼과 텍스트의 대비가 충분한가?
- [ ] 활성 프로필이 색상 외 요소로도 구분되는가?
- [ ] 위험한 동작이 일반 동작과 명확히 구분되는가?
- [ ] 키보드 Focus가 항상 보이는가?
- [ ] 창 배율 변경 시 텍스트가 잘리지 않는가?

### 24.2 프로필 화면

- [ ] 프로필 이름이 가장 먼저 보이는가?
- [ ] 현재 실행 프로필이 정확한 용어로 표시되는가?
- [ ] 실제 계정 이메일을 추정해서 표시하지 않는가?
- [ ] 상태별 주요 버튼 문구가 올바른가?
- [ ] 전환 중 중복 클릭이 차단되는가?
- [ ] 삭제가 실수로 실행되기 어렵게 구성되어 있는가?

### 24.3 오류와 진행

- [ ] 긴 작업 중 진행 상태가 표시되는가?
- [ ] 오류 메시지에 다음 행동이 포함되는가?
- [ ] 기술 상세 정보가 기본 메시지와 분리되어 있는가?
- [ ] 성공과 실패를 색상만으로 전달하지 않는가?
- [ ] Toast가 중요한 확인 모달을 대신하지 않는가?
